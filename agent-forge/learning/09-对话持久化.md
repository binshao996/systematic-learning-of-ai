# 对话持久化

> 基于 Drizzle ORM + PostgreSQL JSONB + SSE 流式通信，从 conversations/messages 表设计、历史消息加载策略、流式输出时同步写库、前端会话列表管理到变量命名冲突处理，完整拆解对话持久化实现。

---

## 1. 设计目标

### 1.1 问题

Agent 测试聊天（Test Chat）需要持久化对话历史：

```
之前: 刷新页面 → 对话消失 → 重新开始
现在: 刷新页面 → 对话还在 → 可继续
```

### 1.2 核心需求

```
1. 多会话 — 一个 Agent 可以有多个对话
2. 历史加载 — 选中对话 → 加载所有消息 → 显示
3. 继续对话 — 发送新消息 → 历史消息作为上下文传给 LLM
4. 流式持久 — SSE 流式输出时，消息同步写入数据库
5. 工具调用记录 — tool_calls JSONB 完整保留
```

## 2. 数据库设计

### 2.1 conversations 表

```sql
CREATE TABLE conversations (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  agent_id    UUID NOT NULL REFERENCES agents(id),
  title       TEXT DEFAULT 'New Chat',
  created_at  TIMESTAMP DEFAULT NOW(),
  updated_at  TIMESTAMP DEFAULT NOW()
);
```

```typescript
export const conversations = pgTable("conversations", {
  id: uuid("id").defaultRandom().primaryKey(),
  agentId: uuid("agent_id").references(() => agents.id).notNull(),
  title: text("title").default("New Chat"),
  createdAt: timestamp("created_at").defaultNow(),
  updatedAt: timestamp("updated_at").defaultNow(),
});
```

### 2.2 messages 表

```sql
CREATE TABLE messages (
  id               UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  conversation_id  UUID NOT NULL REFERENCES conversations(id),
  role             TEXT NOT NULL,          -- 'user' | 'assistant' | 'tool'
  content          TEXT,
  tool_calls       JSONB,                  -- OpenAI tool_calls 格式
  tool_call_id     TEXT,                   -- tool 消息的关联 id
  created_at       TIMESTAMP DEFAULT NOW()
);
```

```typescript
export const messages = pgTable("messages", {
  id: uuid("id").defaultRandom().primaryKey(),
  conversationId: uuid("conversation_id").references(() => conversations.id).notNull(),
  role: text("role").notNull(),
  content: text("content"),
  toolCalls: jsonb("tool_calls"),           // assistant 消息的 tool_calls
  toolCallId: text("tool_call_id"),          // tool 消息的 tool_call_id
  createdAt: timestamp("created_at").defaultNow(),
});
```

### 2.3 表关系

```
agents 1 ──── N conversations 1 ──── N messages

一个 Agent 有多个对话
一个对话有多个消息（user/assistant/tool 交替）
```

## 3. 消息保存策略

### 3.1 写库时机

```
用户发送消息:
  → INSERT messages (role='user', content='...')

每轮 Agent 循环:
  → LLM 返回 tool_calls
    → INSERT messages (role='assistant', content='...', tool_calls=[...])
  → 执行工具
    → INSERT messages (role='tool', content='...', tool_call_id='...')

LLM 最终输出:
  → INSERT messages (role='assistant', content='...')
```

### 3.2 同步写库 + 同步流式

```typescript
// backend/src/services/agent/runtime.ts
export async function runAgent(
  agentId: string, userInput: string,
  controller: ReadableStreamDefaultController,
  conversationId?: string,
) {
  // 1. 创建或加载对话
  let convId = conversationId;
  if (!convId) {
    const [conv] = await db.insert(conversations).values({
      agentId,
      title: userInput.slice(0, 80),
    }).returning();
    convId = conv.id;
  }

  // 2. 加载历史消息作为上下文
  const historyMessages: ChatMessage[] = [];
  if (conversationId) {
    const prevMessages = await db.select().from(messages)
      .where(eq(messages.conversationId, convId!))
      .orderBy(asc(messages.createdAt));

    for (const msg of prevMessages) {
      if (msg.role === "user") {
        historyMessages.push({ role: "user", content: msg.content || "" });
      } else if (msg.role === "assistant") {
        historyMessages.push({
          role: "assistant", content: msg.content || "",
          tool_calls: msg.toolCalls as ChatMessage["tool_calls"],
        });
      } else if (msg.role === "tool") {
        historyMessages.push({
          role: "tool", content: msg.content || "",
          tool_call_id: msg.toolCallId || "",
        });
      }
    }
  }

  // 3. 保存用户消息
  await db.insert(messages).values({
    conversationId: convId!,
    role: "user",
    content: userInput,
  });

  // 4. 构建完整消息队列（system + history + user）
  const messages_arr: ChatMessage[] = [
    { role: "system", content: agent.systemPrompt },
    ...historyMessages,
    { role: "user", content: userInput },
  ];

  // 5. Agent 循环中同步写库
  for (let iteration = 0; iteration < 10; iteration++) {
    const response = await chatCompletion(messages_arr, toolDefs, { ... });

    if (response.tool_calls?.length) {
      // 保存 assistant 消息（含 tool_calls）
      await db.insert(messages).values({
        conversationId: convId!,
        role: "assistant",
        content: response.content || "",
        toolCalls: response.tool_calls,
      });

      for (const tc of response.tool_calls) {
        const result = await executeTool(tc.function.name, tc.function.arguments);
        // 保存 tool 结果消息
        await db.insert(messages).values({
          conversationId: convId!,
          role: "tool",
          content: result,
          toolCallId: tc.id,
        });
      }
    } else {
      // 保存最终 assistant 消息
      await db.insert(messages).values({
        conversationId: convId!,
        role: "assistant",
        content: response.content || "",
      });
      break;
    }
  }
}
```

### 3.3 写库 vs 流式的时间线

```
时间 →

用户发送消息 ──────────────────────────────────────────────
  │
  ├─ INSERT user message (同步，快)
  │
  ├─ SSE: agent_start ──────────────────────────────────────
  │     │
  │     ├─ SSE: thinking ────────────┐
  │     │                            ├─ INSERT assistant (tool_calls)
  │     ├─ SSE: tool_call ──────────┤
  │     ├─ SSE: tool_result ────────┤
  │     │                            ├─ INSERT tool message
  │     ├─ SSE: thinking ────────────┘
  │     │
  │     ├─ INSERT assistant (final)
  │     └─ SSE: agent_output ────────────────────
  │
  └─ SSE: done ─────────────────────────────────────────────
```

写库操作是同步的（await），不阻塞 SSE 流发送。它们交错进行：先写库，再发 SSE。

### 4.2 前端历史加载

```
用户点击侧边栏对话
  │
  ├── loadHistory(agentId, convId)
  │     → GET /api/agents/:id/conversations/:convId/messages
  │     → 将 DB 消息转换为 TraceEvent[]
  │
  └── loadHistoryIntoState(historyEvents, convId)
        → setConversationId(convId)
        → setState({ events: historyEvents, done: true })
```

```typescript
// frontend/src/hooks/use-agent-run.ts
const loadHistory = useCallback(async (agentId: string, convId: string) => {
  const res = await fetch(`${API_URL}/api/agents/${agentId}/conversations/${convId}/messages`);
  const msgs = await res.json();

  const events: TraceEvent[] = [];
  for (let i = 0; i < msgs.length; i++) {
    const msg = msgs[i];
    if (msg.role === "user") {
      events.push({ step: i, type: "user_message" as never, content: msg.content || "" });
    } else if (msg.role === "assistant") {
      events.push({ step: i, type: "agent_output", content: msg.content || "" });
    } else if (msg.role === "tool") {
      events.push({ step: i, type: "tool_result", toolOutput: msg.content || "" });
    }
  }
  return events;
}, []);
```

### 4.3 继续对话时的状态保持

关键 bug 修复：继续对话时不清空 events。

```typescript
// use-agent-run.ts — run() 函数
if (convId) {
  // 继续已有对话 → 追加到现有 events
  setState((s) => ({
    ...s, running: true, done: false, error: null,
    events: [...s.events, userEvent],
  }));
} else {
  // 新建对话 → 从用户消息开始
  setState({
    running: true, done: false, error: null,
    events: [userEvent],
  });
}
```

## 5. 变量命名冲突处理

### 5.1 问题

```typescript
import { messages } from "../../db/schema";  // ← Drizzle messages 表定义

// runtime.ts 函数内部
const messages: ChatMessage[] = [             // ← 局部变量同名
  { role: "system", content: "..." },
];
// ❌ TypeScript: Cannot redeclare block-scoped variable 'messages'
```

### 5.2 解决

```typescript
import { messages } from "../../db/schema";  // Drizzle 表定义

export async function runAgent(...) {
  const messages_arr: ChatMessage[] = [       // 改名为 messages_arr
    { role: "system", content: agent.systemPrompt },
  ];

  // 后续使用 messages_arr 替代原来的 messages
  messages_arr.push({ role: "assistant", ... });
}
```

注意：不需要用 `_unused` 重命名 import，因为 `messages` 表本身在函数内被使用（`db.insert(messages).values(...)`），所以 import 名不能改。改局部变量名是最小改动。

## 7. 设计取舍

```
选择:
  ✅ 自建 conversations/messages 表
  ✅ 同步写库（每步 await）
  ✅ 前端 loadHistory → 转换 DB 行为 TraceEvent

不选:
  ❌ LangGraph Checkpointer 做对话持久化
     理由: 对话是简单的 user↔assistant 消息对，不是图状态。
           Checkpointer 保存的是 StateGraph 的执行状态，
           消息列表更透明且可查询。

  ❌ 异步/批量写库
     理由: 每条消息独立写入保证数据一致性。如果工具执行失败，
          对应的 tool_call 消息已写入，方便调试。

  ❌ 前端 localStorage 做持久化
     理由: 跨设备/浏览器不共享，且数据库查询支持搜索和统计。
```
