# Agent Forge 项目复盘

> 从 0 到 1 构建 Agent 编排平台的完整复盘。涵盖架构决策、技术选择、实现中的坑、做得好的地方、可以改进的地方、以及下一步方向。

---

## 概述

构建了一个可视化的 AI Agent 编排平台，支持 Agent 创建与配置、工具绑定、单 Agent 测试聊天、多 Agent 工作流画布编排、SSE 流式输出、运行历史追溯。这是 AI 全栈学习路线中的项目二，重点是深入掌握 Agent 架构与工作流引擎。

**技术栈：** Next.js 16 + React 19 + ReactFlow | Bun + Hono | PostgreSQL + Drizzle ORM | LangGraph | DeepSeek API

---

## 做得好的地方

### 1. 渐进式架构演进

项目从最简单的单 Agent 问答开始，逐步演进到多 Agent 工作流编排：

```
Phase 1: Agent CRUD + Test Chat (单 Agent SSE 对话)
  → 验证 DeepSeek API 集成 + SSE 流式输出

Phase 2: Tool 系统 (工具定义、绑定、执行循环)
  → 验证 Function Calling 协议

Phase 3: Workflow 编排 (Canvas 画布、多节点、状态传递)
  → 验证 LangGraph 编排能力

Phase 4: UI 打磨 (Markdown 渲染、面板可拖拽、thinking 折叠)
  → 提升使用体验
```

每个阶段都是独立可用的，不会因为下一阶段的复杂度而阻塞当前功能。

### 2. LangGraph 解耦设计

LangGraph 仅用于工作流编排层，不侵入 Agent 运行时。`runAgent()` 和 `runWorkflow()` 是两个独立函数：

- `runAgent()` — 单 Agent 模式，直接 LLM + Tool 循环
- `runWorkflow()` — 多 Agent 模式，通过 StateGraph 编排多个 Agent 节点

这种分离意味着：
- 单 Agent 测试聊天不需要 LangGraph 依赖
- 工作流中的每个节点也可以独立测试
- 未来可以替换 LangGraph 为其他编排引擎，不影响 Agent 核心逻辑

### 3. SSE 三层防护

SSE 流式输出遇到过 Bun 超时 + double-close 的 bug。最终建立了三层防护机制：

| 层 | 位置 | 作用 |
|----|------|------|
| Bun 层 | `idleTimeout: 120` | 减少超时发生 |
| emitSSE 层 | try/catch enqueue | 静默忽略已关闭 stream |
| controller 层 | try/catch close() | 防止 double-close 崩溃 |

每层独立工作，即使上层失效，下层仍能保护服务不崩溃。

### 4. 前端组件复用

`MarkdownContent` 组件被三个地方使用，但只定义一次：

- Test Chat — Agent 测试聊天
- RunConsole — 工作流运行面板
- Run Detail — 运行历史详情页

CSS 也统一在 `globals.css` 的 `.markdown-content` 中，一处修改全局生效。相比 `@tailwindcss/typography` 插件的隐式样式，显式 CSS 更可控。

### 5. 拖拽双协议

Agent 面板同时支持两种拖拽：

- `application/agent` — 携带完整 Agent JSON 数据
- `application/nodetype` — 携带节点类型名（"start"/"end"）

两种数据在同一 `onDrop` 中依次检查处理，互不干扰。扩展新节点类型只需要在面板加一个 draggable div 和画布中对应的检查逻辑。

## 做得不好的地方

### 1. Bun idleTimeout 默认值踩坑

Bun 默认 `idleTimeout: 10` 秒，但文档没有充分强调这对 SSE 长连接的影响。一开始用 Hono 的 `stream()` 方法也遇到了超时问题。最终发现需要在 `Bun.serve()` 层配置 `idleTimeout`。

**教训：** 使用 Bun 作为 SSE 服务器时，第一时间检查 `idleTimeout` 配置。

### 2. 工具调用参数解析脆弱

Calculator 工具的 `arguments` 是 JSON 字符串，但模型有时会生成非标准 JSON：

```
// 正常:
"{\"expression\": \"1+2*3\"}"

// 异常:
"1+2*3"                    // 直接传表达式，没有 JSON
"expression: 1+2*3"        // 非 JSON 格式
```

当前实现通过 try/catch JSON.parse 处理，但对格式错误没有做 fallback 尝试。更好的做法是增加参数清洗层。

### 3. WorkflowNode type 字段后加

最初 `WorkflowNode` 只有 `agentId`，后来才加 `type` 字段。这导致：
- 旧工作流数据没有 `type`，加载时需要用默认值 "agent" 兜底
- 前端 `nodeType` 和后端 `type` 需要双向映射

**教训：** 设计 JSONB 字段时，第一时间加入 `type` 这样的区分字段，即使最初只有一种类型。

### 4. 缺少集成测试

没有端到端测试覆盖以下流程：
- 创建 Agent → 发送消息 → 收到 SSE 事件 → 渲染输出
- 创建工作流 → 拖入节点 → 连线 → 保存 → 加载 → 运行

所有验证通过手动操作完成。对于工作流这种复杂交互，自动化测试能防止重构引入回归。

### 5. 前端工作流编辑器缺少撤销/重做

ReactFlow 的 `onNodesChange` 和 `onEdgesChange` 直接修改 state，没有历史记录。用户误删节点或错误连线后无法撤销。

ReactFlow 文档中没有内置 undo/redo，需要用 `useNodesState` 的底层 API 配合历史栈实现。

## 关键决策

### 为什么后端用 JSONB 存 nodes/edges 而不是关系表？

```
方案 A: JSONB (当前)
  workflows.nodes = [{"id":"a","type":"agent",...}, ...]
  workflows.edges = [{"id":"e1","source":"a","target":"b"}, ...]

方案 B: 关系表
  workflow_nodes (id, workflow_id, type, agent_id, position_x, position_y, ...)
  workflow_edges (id, workflow_id, source_node_id, target_node_id, ...)
```

**选择 JSONB 的理由：**
- 工作流节点/边的查询永远是一起读的（加载整个工作流），不存在 "只查某个节点" 的场景
- 节点位置 (x, y) 是画布坐标，JSON 比多个列更自然
- Drizzle ORM + PostgreSQL JSONB 支持良好
- 无需 join，一次查询拿到完整工作流

### 为什么没有用 LangChain Agent 而自己写 Agent 循环？

```
LangChain Agent 方案:
  createReactAgent() → 自动管理 tool calling 循环

自写循环方案:
  for (let i = 0; i < 10; i++) {
    response = chatCompletion(messages, tools)
    if (tool_calls) execute → push → continue
    else break
  }
```

**选择自写循环的理由：**
- 循环逻辑很简单（~20 行），不需要框架抽象
- 完全控制 SSE 事件推送时机
- 不引入 LangChain 的 AgentExecutor 等重依赖
- 调试时可以直接看 messages 数组的累积过程

### 为什么前端用 react-markdown 而不是 @tailwindcss/typography？

最初尝试用 `@tailwindcss/typography` 的 `prose` 类渲染 Markdown，但：
- `@tailwindcss/typography` v0.5.x 与 Tailwind v4 的 `@plugin` 加载有兼容性问题
- `prose` 类的具体样式是黑盒，想改 pre 背景色时需要查文档

最终选择显式 `.markdown-content` CSS（~120 行），所有元素样式一目了然，修改即生效。

## 可以改进的地方

### 1. 安全性

**Code 节点使用 `new Function()` 执行代码：**
```
风险: 任意 JS 代码执行
当前: 无沙箱，无权限控制
改进: Web Worker 沙箱 / vm2 隔离 / 限制执行的 API 白名单
```

**API 无鉴权：**
```
风险: 任何知道端口的人都能调 API
当前: 无 token / session 验证
改进: JWT 或 API Key 中间件
```

### 2. 性能

**大量事件时 React 渲染卡顿：**
```
当前: 每个 SSE 事件触发 setState → 全量 events 数组复制
改进: 虚拟列表 (react-virtuoso) 只渲染可见事件
```

**DeepSeek API 调用无超时：**
```
当前: fetch 无 AbortSignal timeout，依赖 Bun idleTimeout
改进: AbortSignal.timeout(30000) 或 Promise.race
```

### 3. 用户体验

**工作流编辑器缺少：**
- 撤销/重做 (Ctrl+Z / Ctrl+Y)
- 节点复制粘贴 (Ctrl+C / Ctrl+V)
- 多选节点 (Shift+Click / 框选)
- 自动布局 (Dagre 算法排列节点)

**Code 节点缺少：**
- 代码编辑器（Monaco / CodeMirror 集成）
- 语法高亮
- 执行结果预览

### 4. 可观测性

**缺少日志和监控：**
- Agent 调用耗时分布
- 工具调用失败率
- SSE 连接断开频率
- 模型 token 用量统计

## 下一步方向

### 1. Agent 模板市场
预设常用 Agent 模板（代码审查、文案润色、数据翻译），用户一键创建。

### 2. 对话历史持久化
将 Test Chat 的对话历史存入 DB，支持历史回看和继续对话。

### 3. 工作流并行执行
LangGraph 支持并行节点。多个入度为 0 的节点从 START 并行起跑，可大幅提升执行效率。

### 4. Human-in-the-loop
在工作流中加入 "暂停等待人工审批" 节点，适用于需要人工判断的关键步骤。

### 5. 自定义工具
用户可以创建自定义工具（HTTP API 调用、Python 脚本执行），而不限于内置工具。

## 总结

```
Agent Forge = Agent 配置 + 工具系统 + 单 Agent 对话 + 多 Agent 工作流 + SSE 流式输出

核心收获:
  1. LangGraph StateGraph 是多 Agent 编排的优秀抽象
  2. SSE 流式通信需要三层防护机制
  3. 前端 Canvas 编辑器需要为未来扩展预留 type 字段
  4. react-markdown + 显式 CSS 比 Tailwind Typography 插件更可控
  5. 工具调用参数格式需要健壮的解析和 fallback

总代码量: ~2000 行 (后端 ~800 + 前端 ~1200)
```
