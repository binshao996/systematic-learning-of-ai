# 并行工作流执行

> 基于 LangGraph StateGraph + SSE 流式事件，从并行触发条件、parallel_start/parallel_end 事件设计、前端多节点高亮、ReactFlow MiniMap 动态着色到并发安全考量，完整拆解并行工作流执行实现。

---

## 1. 并行执行概述

### 1.1 什么是并行执行

```
串行:
  Start → Agent A → Agent B → End     A 等 B，总耗时 = A + B

并行:
         ┌→ Agent A ─┐
  Start ─┤            ├→ End           A、B 同时跑，总耗时 = max(A, B)
         └→ Agent B ─┘
```

当多个节点都从 START 出发（或从同一个上游节点出发），它们之间没有数据依赖，可以同时执行。

### 1.2 LangGraph 的并行支持

LangGraph StateGraph 原生支持并行：
- 多个 entry nodes 连接到 START → 自动并行
- 节点的状态通过 reducer 合并
- 拓扑排序自动处理 fan-in（等待所有并行节点完成后再继续）

## 2. 并行检测机制

### 2.1 Entry Nodes 计算

```typescript
// buildGraph() 中的边缘分析
const incomingCount = new Map<string, number>();

for (const edge of edges) {
  incomingCount.set(edge.target, (incomingCount.get(edge.target) || 0) + 1);
}

// 没有入边的节点 = entry node
const entryNodes = nodes.filter((n) => !incomingCount.has(n.id));

if (entryNodes.length > 1) {
  // 并行执行
  for (const n of entryNodes) graph.addEdge(START, n.id);
} else if (entryNodes.length === 1) {
  graph.addEdge(START, entryNodes[0].id);
}
```

```
示例 workflow:
  nodes: [Start(id:a), Agent(id:b), Agent(id:c), End(id:d)]
  edges: [a→b, a→c, b→d, c→d]

  incomingCount: { b:1, c:1, d:2 }
  entryNodes: [a]           ← 只有 Start 没有入边
  hasOutgoing: { a, b, c }  ← d 没有出边，自动连 END
```

```
并行 workflow:
  nodes: [Agent(id:a), Agent(id:b), End(id:c)]
  edges: [a→c, b→c]

  incomingCount: { c:2 }
  entryNodes: [a, b]        ← 两个节点没有入边 → 并行
  hasOutgoing: { a, b }     ← c 没有出边，自动连 END
```

### 2.2 更广泛的并行场景

```
LangGraph 支持的不只是 entry nodes 并行：

1. 多 entry nodes:
   START → [A, B] → END              ← 当前利用

2. 单源 fan-out:
   Node X → [A, B] → END             ← LangGraph 原生，无需特殊处理

3. 条件 fan-out:
   Node X → ConditionalEdges → [A, B] ← 基于条件路由，非并行
```

Agent Forge 当前利用场景 1。场景 2 如果出现，LangGraph 自动并行无需额外处理。场景 3 是条件分支，不涉及并行。

## 3. SSE 事件协议

### 3.1 并行事件类型

```typescript
// backend/src/services/agent/stream.ts
export type SSEEventType =
  | "agent_start"
  | "thinking"
  | "tool_call"
  | "tool_result"
  | "agent_output"
  | "node_start"
  | "node_output"
  | "error"
  | "done"
  | "parallel_start"    // ← 新增：并行区域开始
  | "parallel_end";     // ← 新增：并行区域结束

export interface SSEEvent {
  // ... 原有字段
  branchCount?: number;  // ← 并行分支数
}
```

### 3.2 并行事件流示例

```
两个 Agent 并行执行的 SSE 流:

parallel_start { branchCount: 2 }
node_start     { agentId: "agent_a" }
agent_start    { agentId: "agent_a", agentName: "Code Reviewer" }
node_start     { agentId: "agent_b" }
agent_start    { agentId: "agent_b", agentName: "Copy Editor" }
thinking       { agentId: "agent_a", content: "..." }
thinking       { agentId: "agent_b", content: "..." }
tool_call      { agentId: "agent_a", toolName: "web_search" }
tool_result    { agentId: "agent_a", ... }
agent_output   { agentId: "agent_a", content: "Review result..." }
agent_output   { agentId: "agent_b", content: "Edit result..." }
parallel_end
done
```

注意：两个分支的事件是**交错**的（真实并发），不是 A 完成后再 B。

### 3.3 事件触发位置

```typescript
// backend/src/services/workflow/runner.ts — buildGraph()
const entryNodes = nodes.filter((n) => !incomingCount.has(n.id));
const isParallel = entryNodes.length > 1;

if (isParallel) {
  // 在连接边之前发射 parallel_start
  const parallelEvent: SSEEvent = {
    type: "parallel_start",
    step: 0,
    branchCount: entryNodes.length,
  };
  emitSSE(controller, parallelEvent);
  logEvent(parallelEvent);
}

// ... 连接边、编译图、invoke ...

if (isParallel) {
  // 在 invoke 完成后发射 parallel_end
  emitSSE(controller, {
    type: "parallel_end",
    step: nodes.length,
  });
}
```

## 4. 前端多节点高亮

### 4.1 activeNodeIds 追踪

```
单个节点执行:
  activeNodeIds = Set { agentId_a }  → 高亮 1 个节点

并行执行:
  activeNodeIds = Set { agentId_a, agentId_b }  → 同时高亮 2 个节点
```

```typescript
// frontend/src/hooks/use-workflow-run.ts

if (event.type === "agent_start" || event.type === "node_start") {
  setState((s) => {
    const newActive = new Set(s.activeNodeIds);
    newActive.add(event.agentId);       // 添加
    return {
      ...s,
      events: [...s.events, event],
      activeNodeIds: newActive,         // 新 Set 触发重渲染
    };
  });
} else if (event.type === "agent_output" || event.type === "node_output") {
  setState((s) => {
    const newActive = new Set(s.activeNodeIds);
    newActive.delete(event.agentId);    // 移除
    return { ...s, events: [...s.events, event], activeNodeIds: newActive };
  });
} else if (event.type === "parallel_end") {
  setState((s) => ({
    ...s, activeNodeIds: new Set(),     // 并行结束，清空
  }));
}
```

### 4.2 Canvas 节点着色

```typescript
// frontend/src/app/workflows/[id]/page.tsx
const activeNodes = nodes.map((n) => {
  const isRunning = running && activeNodeIds.has(n.data.agentId || n.id);
  const isCompleted = done || events.some(
    (e) => (e.type === "agent_output" || e.type === "node_output") &&
      (e.agentId === n.data.agentId || e.agentId === n.id)

});
```

## 5. 状态合并

### 5.1 Annotation Reducer

LangGraph 并行节点通过 reducer 合并状态：

```typescript
const WorkflowState = Annotation.Root({
  agentOutputs: Annotation<Record<string, string>>({
    reducer: (current, update) => ({ ...current, ...update }),
    default: () => ({}),
  }),
  executionLog: Annotation<Array<{ nodeId: string; agentName: string; output: string }>>({
    reducer: (current, update) => [...current, ...update],
    default: () => ([]),
  }),
});
```

```
并行执行时的状态合并:

  Agent A 返回: { agentOutputs: { "a": "Result A" }, executionLog: [{...}] }
  Agent B 返回: { agentOutputs: { "b": "Result B" }, executionLog: [{...}] }

  合并后:
    agentOutputs: { "a": "Result A", "b": "Result B" }  ← 对象合并
    executionLog: [{...}, {...}]                          ← 数组拼接
```

### 5.2 End 节点汇总

```typescript
// runner.ts — End 节点处理
graph.addNode(node.id, (state: typeof WorkflowState.State) => {
  // 所有并行分支的 agentOutputs 合并为一个字符串
  const output = Object.values(state.agentOutputs).join("\n\n") || state.input;
  return { agentOutputs: { [node.id]: output } };
});
```

## 6. 并发安全考量

### 6.1 SSE 写入安全

```typescript
// emitSSE 通过 ReadableStreamDefaultController.enqueue 写入
// Web Streams API 规范：enqueue 是线程安全的
export function emitSSE(controller: ReadableStreamDefaultController, event: SSEEvent) {
  try {
    const data = JSON.stringify(event);
    controller.enqueue(new TextEncoder().encode(`event: ${event.type}\ndata: ${data}\n\n`));
  } catch {
    // Controller 已关闭（客户端断开/超时）— 忽略
  }
}
```

### 6.2 跟踪事件写库

```typescript
// logEvent 使用 .execute().catch(() => {}) 异步写库
// 不 await，避免阻塞图执行
const logEvent = (event: SSEEvent) => {
  allEvents.push(event);
  db.update(runs)
    .set({ traceEvents: allEvents })
    .where(eq(runs.id, runId))
    .execute()
    .catch(() => {});
};
```

潜在问题：并行节点的 logEvent 可能交错调用，导致 allEvents 中的事件顺序不完全准确。当前实现用单数组 push，事件顺序取决于 JS 事件循环的 microtask 调度。对于观测用途（trace 记录），可接受。

### 6.3 数据库写入

并行节点各自写自己的 trace events，但最终 `runWorkflow` 通过 `logEvent` 闭包共享 `allEvents` 数组。JS 的单线程模型保证数组 push 是安全的。

## 7. 当前限制

```
1. 无显式 fan-out 节点
   → 并行只能通过 START → 多节点实现
   → 如需中间节点 fan-out，需添加 Send/Receive API

2. 无分支 ID 追踪
   → 并行事件交错，无法区分属于哪个分支
   → 如需分支级追踪，需在 SSEEvent 中添加 branchId 字段

3. 无并行超时控制
   → LangGraph 不提供内置超时
   → 需在节点函数中手动 AbortSignal 传递

4. MiniMap 着色粒度
   → 当前基于 agentId 匹配，非节点 ID
   → 同一 Agent 被多次使用时可能误着色
```
