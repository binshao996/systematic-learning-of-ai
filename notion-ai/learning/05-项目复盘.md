# 项目一复盘：Notion AI 知识平台

## 概述

构建了一个类似 Notion 的知识管理平台，包含完整的 RAG 管道，跨越 5 个阶段共 30 个任务。这是 AI 全栈学习路线中的项目一，重点是深入掌握 RAG（Retrieval Augmented Generation，检索增强生成）。

**技术栈：** Next.js 16 + TipTap | Bun + Hono | PostgreSQL + pgvector | Qdrant | DeepSeek API | MinIO

## 做得好的地方

### 1. 编码之前先进行结构化学习（阶段一）
在编写任何应用代码之前，花两周时间学习 RAG、DeepSeek API、Qdrant 以及文档解析的基础知识，结果证明这是值得的。实验性练习脚本（`learning/*.ts`）提供了一个安全的沙箱环境，让我们在不涉及应用层复杂性的前提下理解这些 API。

### 2. 子代理驱动的开发
每个任务派发全新的子代理，配合两阶段评审（需求符合度 → 代码质量），使实现保持快速和专注。机械性任务（CRUD 路由、UI 组件）由较小的模型高效完成，而复杂任务（分块策略、RAG 引擎）则获得完整的上下文推理。

### 3. 关键模块采用 TDD
分块模块是应用 TDD 的正确位置。三种策略（固定、语义、递归）以及边界情况（重叠导致的无限循环、位置 0 缺少标题）在测试中被发现，避免了在生产环境中成为 bug。7 个分块测试发现了 2 个真实的 bug，这些问题在生产环境中调试会非常痛苦。

### 4. 清晰的服务边界
将摄入管道分离为 parser → chunker → indexer → pipeline，将 RAG 引擎分离为 retriever → generator → engine，创建了清晰的接口。每个服务都可以独立测试，并且具有单一职责。集成测试（任务 27）在不依赖所有服务运行的情况下验证了组合的正确性。

### 5. DeepSeek API 兼容性
DeepSeek 兼容 OpenAI 的 API 使得集成非常简单。聊天补全端点（`/v1/chat/completions`）、嵌入端点（`/v1/embeddings`）和流式传输（SSE）都可以通过标准的 fetch 调用使用。无需 SDK — TypeScript 类型 + fetch 已足够。

## 做得不好的地方

### 1. 环境相关的问题
Bun 的 AVX 要求导致开发机器上的二进制兼容性问题，需要使用 `bun-darwin-x64-baseline` 构建。lightningcss 原生模块也无法解析。这些平台特定问题浪费了时间，并且没有通用的修复方法 — 它们依赖于具体的运行环境。

### 2. Next.js 16 破坏性变更
项目使用了 Next.js 16.2.6，相比文档更为普遍覆盖的 Next.js 14/15，存在破坏性变更。`@base-ui/react` Dialog API（此版本中 shadcn/ui 使用的）使用了 `render` prop 而不是 `asChild`，导致上传对话框中持续存在类型错误。该版本的文档非常稀少。

### 3. 缺乏真正的集成测试
集成测试在 Docker 服务或 DeepSeek API 不可用时会优雅跳过。这意味着"真实"的端到端流程（文件上传 → DeepSeek 实时嵌入 → Qdrant 搜索 → AI 聊天）从未以自动化方式完整测试过。拥有 Docker 服务的 CI 管道可以解决这个问题，但这超出了 MVP 的范围。

### 4. 过多并行任务导致竞态条件
并行派发任务 5/6/7（学习笔记）导致两个任务的文件因 git 竞态条件而落入了同一个提交。文件内容都是正确的，但提交历史被搞乱了。教训：即使任务操作的是不同文件，对于写入同一目录的任务也必须严格串行化。

## 关键技术决策

### 为什么选择 DeepSeek API 而不是 OpenAI？
- **成本：** DeepSeek 每 token 费用显著更低
- **中文支持：** 翻译功能中中文文本处理表现更好
- **API 兼容性：** OpenAI 兼容端点意味着后续切换时零迁移成本
- **嵌入维度：** 1536 维向量，与 OpenAI text-embedding-ada-002 一致

### 为什么选择 Qdrant 而不是 Pinecone/Weaviate？
- **自托管：** Docker Compose 本地开发，不依赖云服务
- **过滤功能：** 基于 payload 的过滤支持按文档搜索范围
- **性能：** Rust 原生，轻松处理项目规模
- **学习价值：** 运行自己的向量数据库比使用托管服务能学到更多

### 为什么选择语义分块作为默认策略？
- 在块连贯性和简洁性之间取得最佳平衡
- 按标题分割保留文档结构，便于引用
- 固定大小分块（含重叠）适用于密集技术文档
- 递归分块处理边界情况（非常长的无标题节）

### 为什么选择 Bun + Hono 而不是 Node + Express？
- **性能：** Bun 启动时间近乎即时的
- **TypeScript：** 无需 ts-node 或 tsx 的一流 TS 支持
- **Hono：** 轻量级，内建 Zod 验证中间件，Web 标准 API
- 更简单的部署模型（单一二进制文件）

### 为什么选择 TipTap 而不是其他编辑器？
- **可扩展性：** 基于 ProseMirror，每个功能都是一个扩展
- **自定义节点：** 后续可以添加引用高亮、AI 生成块等
- **React 集成：** `useEditor` hook 与 React 组件模型自然契合
- **活跃的生态：** 提供 StarterKit、Placeholder、Highlight 等扩展

## RAG 性能说明

评估框架（`services/eval/evaluator.ts`）衡量：
- **忠实度（Faithfulness）：** 答案在多大程度上基于检索到的块
- **相关性（Relevance）：** 检索到的块在多大程度上匹配查询
- **延迟（Latency）：** 端到端响应时间
- **Token 用量：** 输入 + 输出 token 计数

关键观察（未运行完整评估 — 需要 DeepSeek API key）：
- 嵌入质量是检索质量中最重要的因素
- 对于大多数文档，块大小比分块策略更重要
- 引用反馈循环（点赞/踩）对于随时间改善检索至关重要
- 流式传输对于感知性能至关重要 — 用户可以实时看到 token 的生成

## 最大的收获

### 1. RAG 是一个管道，而非一个功能
每个阶段（解析、分块、嵌入、索引、检索、生成）都会影响其他每一个阶段。糟糕的分块策略产生糟糕的嵌入，糟糕的嵌入产生糟糕的检索，糟糕的检索产生糟糕的回答。管道必须端到端评估，不能逐个组件评估。

### 2. 嵌入是桥梁
嵌入模型是 RAG 的"罗塞塔石碑" — 它将文档和查询都翻译到同一个向量空间中。如果嵌入模型不理解你的内容领域，下游的一切都无法正常工作。DeepSeek 的嵌入模型对英文和中文的处理都很好，这对翻译功能至关重要。

### 3. 引用是一项产品功能
仅仅生成答案是不够的 — 用户需要能够验证来源。`[chunk:UUID]` 引用标记模式 + 可点击的引用链接建立了信任。反馈循环（👍/👎）将用户行为转化为质量信号。

### 4. SSE 流式传输改变用户体验
聊天流式传输的 Server-Sent Events 让 AI 感觉响应更快。打字指示器 + 增量文本显示创造了一种同步响应无法比拟的对话感。但 SSE 解析是脆弱的 — 流式聊天 hook 必须处理不完整的块、`[DONE]` 标记和连接中断。

### 5. TypeScript + Zod = 全栈类型安全
在启动时用 Zod 验证环境变量、在路由处理中验证请求体、在前端验证 API 响应，创建了从配置到 UI 的类型安全管道。一个 schema 变更可传播到各个层面。

## 阶段五 — AI 写作助手（编辑器内联 AI）
（阶段五是最新的阶段，在原有四个阶段基础上新增。）

在阶段五中，我们用内联到编辑器中的 AI 功能取代了独立的聊天面板，通过 TipTap 自定义扩展实现了深度集成。以下是该阶段的关键产出：

- **替代独立 ChatPanel：** 不再在编辑器旁设置独立的聊天面板，而是通过 TipTap 扩展将 AI 内联到编辑器中。
- **自定义 AIBlock 节点：** 一个内联 AI 交互块，支持 write/qa 两种模式。实现了 input → loading → done 状态机。支持对话历史记录查看，以及 Keep / Retry / Discard / Continue 四种操作按钮。
- **自定义 AICommand 扩展：** 提供了 `insertAIBlock()` 和 `insertAIQA()` 两条命令供编辑器调用。
- **AIBubbleMenu 浮动菜单：** 选中文本后显示浮动菜单，提供 8 种 AI 操作（improve、rewrite、summarize、translate-zh/en、longer、shorter、tone），通过 SSE 流式传输预览结果，并支持 Replace（替换原文）和 Insert below（在下方插入）两种处理方式。
- **键盘快捷方式：** 在空段落中按空格键触发 AI Write 模式；输入 `/ai` 或 `/aik` 后按空格/回车触发内联 AI 块。
- **共享 SSE 客户端（`stream-client.ts`）：** 抽取了共用的 SSE 解析逻辑，供 chat、AIBlock、AIBubbleMenu 三处使用。
- **Markdown 渲染（`markdown.ts`）：** 自定义转换器，用于显示 AI 响应内容，支持标题、列表、代码块、粗体/斜体、链接。
- **移除 SearchDialog 组件：** 编辑页面不再显示搜索对话框和聊天面板。
- **首页服务端重定向：** 首页自动重定向到第一篇文档（匹配 Notion 的产品行为）。
- **侧边栏改进：** 重命名为 "bin notion"，新增鼠标悬停时出现的 "..." 删除上下文菜单。
- **工具栏按钮修复：** 使用原生 button 元素 + `onMouseDown preventDefault` 模式修复了工具栏按钮的问题。
- **显式 TipTap 编辑器内容样式：** 当 `@tailwindcss/typography` 插件 CSS 未生成时，为编辑器添加了显式的内容样式（标题、列表、代码块）。

## 项目二中需要改进的地方

### 1. 从第一天起就使用 LangChain
项目一手动构建了 RAG 管道（对学习有益）。项目二应使用 LangChain.js + LangGraph.js 进行 agent 编排、工具调用和工作流管理。手写方式已经教会了我们基础知识，现在是学习框架的时候了。

### 2. 更好的流式传输架构
当前的 SSE 实现是裸 fetch + ReadableStream 解析。对于项目二的多 agent 工作流，需要考虑：
- 结构化的流式事件（agent 步骤、工具调用、结果）
- 对长时间运行的 agent 链支持中止/重试
- 对快速事件流的背压处理

### 3. 真正的集成测试
在 CI 中设置 Docker 服务，并使用实际的 DeepSeek API 运行完整的端到端测试。评估框架已经存在 — 应该自动化运行。

### 4. 从项目开始就加入可观测性
从第一天起添加结构化日志、请求 ID 和延迟追踪。多 agent 工作流比单个 RAG 查询更难调试。至少需要：agent 链的 OpenTelemetry tracing。

### 5. 先设计后编码
头脑风暴 → 规格 → 方案 → 实施的工作流程效果很好。对于项目二更复杂的架构（agent 运行时、工具注册表、工作流图），在规范阶段投入更多时间，在编写代码前确保抽象层设计正确。

### 6. 避免对共享目录的并行写入
任务 5/6/7 中的竞态条件是可以避免的。对于项目二，即使任务操作不同文件，也必须串行化任何写入同一目录的任务。

## 结语

项目一证明了使用 DeepSeek API + 开源基础设施构建生产质量 RAG 系统是可行的。核心洞察是：RAG 本质上是一个系统集成问题 — AI 部分（嵌入、生成）是 API 调用，而工程部分（解析、分块、索引、流式传输、UI）才是大部分工作所在。

"结构化学习 → 构建 → 复盘"的循环运转良好。项目二（使用 LangChain 的多 agent 工作流）将在 RAG 基础之上，引入 agent 编排、工具调用和工作流管理。
