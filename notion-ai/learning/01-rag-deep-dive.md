# RAG Deep Dive — Structured Learning Notes

## 1. Embedding Fundamentals

### What is an Embedding Vector?

An embedding is a dense numerical representation of text (or any data) in a high-dimensional vector space. It is generated by passing input text through a neural network (typically a Transformer-based encoder) that maps semantic meaning to spatial coordinates. Sentences with similar meanings cluster together in this space.

```
[Tokenization] -> [Transformer Encoder] -> [Mean Pooling / CLS token] -> [Dense Vector]
     "What is RAG?"              BERT model              768-dim float array
```

### Cosine Similarity for Retrieval

Given a query embedding `q` and a document embedding `d`, similarity is:

```
cosine_similarity(q, d) = (q . d) / (||q|| * ||d||)
```

- Range: `[-1, 1]` in theory, but for text embeddings typically `[0, 1]`
- Only the **direction** matters (magnitude is normalized away)
- Pre-normalize all vectors to unit length so dot product == cosine similarity

### Embedding Model Tradeoffs

| Factor | Low Dim (384-512) | High Dim (1024-4096) |
|---|---|---|
| Storage cost | Low | High |
| Retrieval speed | Fast | Slower |
| Semantic fidelity | Adequate for simple tasks | Better for nuanced/domain tasks |
| Example | `text-embedding-3-small` | `text-embedding-3-large` |

| Factor | Monolingual | Multilingual |
|---|---|---|
| Accuracy | Higher on single language | Lower per-language (capacity spread) |
| Use case | English-only apps | Global products |
| Example | `BGE-base-en` | `DeepSeek-Embedding` |

### DeepSeek Embedding API

- **Model**: `deepseek-embedding` (free tier available)
- **Dimensions**: 1024 (note: smaller than ada-002's 1536)
- **Max tokens**: 4096 per input
- **Language**: Strong multilingual support (Chinese, English, others)
- **Endpoint**: `POST https://api.deepseek.com/v1/embeddings`
- **Input format**:

```json
{
  "model": "deepseek-embedding",
  "input": ["document text here"]
}
```

**Response**:

```json
{
  "data": [
    {
      "embedding": [0.012, -0.034, ...],
      "index": 0
    }
  ],
  "model": "deepseek-embedding",
  "usage": {
    "prompt_tokens": 8,
    "total_tokens": 8
  }
}
```

> **Note**: DeepSeek embeddings output 1024 dimensions. If your chunking produces very short fragments, consider concatenating related chunks or padding input to avoid sparse representations.

## 2. Chunking Strategies

### Fixed-Size Chunking

Split text into N-token chunks regardless of content boundaries.

```typescript
function fixedSizeChunks(text: string, chunkSize: number, overlap: number): string[] {
  const tokens = tokenize(text);  // or simply split by whitespace
  const chunks: string[] = [];
  for (let i = 0; i < tokens.length; i += chunkSize - overlap) {
    chunks.push(tokens.slice(i, i + chunkSize).join(" "));
  }
  return chunks;
}
```

| Pros | Cons |
|---|---|
| Simple, predictable | May cut sentences/paragraphs in half |
| Fast to implement | No semantic coherence |
| Deterministic | Poor retrieval for mid-chunk concepts |

**When to use**: Quick prototypes, homogeneous text (e.g., log entries).

### Semantic Chunking

Split on natural boundaries: paragraphs, sections, headings.

```typescript
function semanticChunks(text: string, maxTokens: number): string[] {
  const paragraphs = text.split(/\n\s*\n/);
  const chunks: string[] = [];
  let current: string[] = [];

  for (const para of paragraphs) {
    if (tokenCount([...current, para].join("\n")) > maxTokens && current.length > 0) {
      chunks.push(current.join("\n"));
      current = [para];
    } else {
      current.push(para);
    }
  }
  if (current.length > 0) chunks.push(current.join("\n"));
  return chunks;
}
```

| Pros | Cons |
|---|---|
| Preserves meaning units | May create uneven chunk sizes |
| Better retrieval relevance | Harder to tune max-token limit |
| Human-readable chunks | More complex logic |

**When to use**: Well-structured documents (articles, docs, books).

### Recursive Chunking

Try multiple separators in order and recursively split until within target size.

```
Priority: \n\n (double newline) > \n (newline) > . (sentence) > " " (word)
```

Algorithm:
1. Try splitting by `\n\n`. If chunks are still too large, take the largest chunk.
2. Split that chunk by `\n`. Repeat until within range.
3. Fall through to sentence split, then word split.

Used by LangChain's `RecursiveCharacterTextSplitter`. Best general-purpose chunker.

### Sliding Window Overlap

Overlap ensures information at chunk boundaries isn't lost.

- **Why**: A sentence spanning two chunks becomes unreachable if split mid-sentence.
- **How much**: 10-20% of chunk size is typical.
  - 512-token chunks -> 50-100 token overlap
  - 1024-token chunks -> 100-200 token overlap
- **Tradeoff**: More overlap = more storage + slower retrieval, but higher recall.

### How Chunk Size Affects Retrieval

| Chunk Size | Precision | Recall | Context Quality |
|---|---|---|---|
| Too small (32-64) | High (exact matches) | Low (misses context) | Poor (no surrounding info) |
| Optimal (256-512) | Balanced | Balanced | Good |
| Too large (1024+) | Low (noise in results) | High (lots of matches) | Poor (distracts LLM) |

General rule: **match chunk size to expected answer length**. For Q&A over paragraphs, 256-512 tokens works well. For summarization of sections, 1024+ may be appropriate.

### Metadata Preservation

Critical for traceability and source attribution:

```typescript
interface Chunk {
  id: string;
  text: string;
  metadata: {
    source: string;          // file name / URL
    heading: string;         // nearest heading
    position: number;        // order in document
    start_char: number;      // char offset in original
    chunk_size_tokens: number;
  };
  embedding: number[];
}
```

- **Headings**: Track the nearest section heading so the LLM can cite sources.
- **Position**: Preserve document order for re-ranking tiebreakers.
- **Source**: Required for citation and debugging.

## 3. Retrieval Approaches

### Dense Retrieval (Vector Similarity)

Query and documents are embedded into the same vector space. Search = nearest neighbor by cosine similarity.

```
retrieve(query, docs, k=5):
  q_vec = embed(query)
  scores = [cosine_sim(q_vec, d.vec) for d in docs]
  top_k = docs.argsort(scores, descending)[:k]
  return top_k
```

- **Strengths**: Captures semantic similarity, works with synonyms, handles paraphrasing.
- **Weaknesses**: Requires good embeddings, high dimensional storage, cold-start on new domains.

### Sparse Retrieval (BM25 / Keyword)

BM25 is a bag-of-words ranking function that scores documents by term frequency and inverse document frequency:

```
BM25(d, q) = sum over tokens t in q of IDF(t) * (TF(t,d) * (k1 + 1)) / (TF(t,d) + k1 * (1 - b + b * |d|/avgdl))
```

Where:
- `k1 = 1.2` (term frequency saturation)
- `b = 0.75` (length normalization)
- `avgdl` = average document length in the corpus

| Dense | Sparse |
|---|---|
| Understands meaning | Matches keywords exactly |
| Needs GPU for embedding | Needs only inverted index |
| Slow on 10M+ docs (w/o ANN) | Fast at any scale |
| Fails on rare terms | Excels at rare/technical terms |

### Hybrid Search

Combine dense and sparse scores with weighted sum:

```typescript
function hybridScore(denseScore: number, sparseScore: number, alpha: number): number {
  // Normalize both to [0, 1] first (min-max or rank normalization)
  return alpha * denseScore + (1 - alpha) * sparseScore;
}
```

- `alpha = 0.7`: dense-heavy, good for semantic search
- `alpha = 0.3`: keyword-heavy, good for technical/domain search  
- `alpha = 0.5`: balanced general purpose

**Rank fusion** (Reciprocal Rank Fusion) is an alternative that avoids score normalization:

```
RRF(doc) = sum over rank r of 1 / (k + r)
```

Where `k = 60` (constant). Combines ranked lists without caring about raw scores.

### Reranking

After retrieving top-k (e.g., 20-50) candidates with a fast retriever, a slower but more accurate **cross-encoder reranker** re-scores them.

```
Fast retriever (cosine/BM25)  ->  top-50 candidates
Cross-encoder reranker         ->  top-5 final results
```

- **Why**: Embedding models produce one vector per document — they lose the fine-grained query-doc interaction. Cross-encoders process `(query, doc)` pairs jointly, yielding much better relevance scores.
- **When**: Always, if latency permits. Adding a reranker is the single highest-impact retrieval improvement.
- **Models**: `BAAI/bge-reranker-v2-m3`, `Cohere rerank`, `DeepSeek-Reranker`.
- **Cost**: O(k) cross-encoder calls per query. If `k=50`, that's 50 forward passes.

## 4. RAG Pipeline Patterns

### Naive RAG

The simplest form: retrieve, then augment, then generate.

```
User Query
    |
    v
[Embedding] -> [Vector Search] -> top-k chunks
    |
    v
[Prompt Assembly]: System prompt + "Context:\n" + chunks + "\nQuery:\n" + query
    |
    v
[LLM Generate] -> Answer
```

**Limitations**:
- No query understanding or rewriting
- Retriever gets one shot — no feedback loop
- Context window may overflow with irrelevant chunks
- No mechanism to handle missing information gracefully

### Advanced RAG: Query Transformation

Improve the query before retrieval:

```typescript
// Query rewriting: LLM rewrites the user's ambiguous query
const rewrittenQuery = await llm.complete(
  `Given the question, rewrite it to be more specific and searchable:\nQuestion: ${query}`
);

// Query expansion: generate multiple search queries
const queries = await llm.complete(
  `Generate 3 different search queries for:\n${query}`
);

// HyDE: generate a hypothetical answer, then use that for retrieval
const hypotheticalDoc = await llm.complete(`Answer: ${query}`);
const hydeEmbedding = await embed(hypotheticalDoc);
```

### Multi-Step Retrieval

**Iterative retrieval**: Retrieve, generate partial answer, identify gaps, retrieve more.

```
[Query] -> [Retrieve] -> [Generate partial] 
    ^                           |
    |--- [Identify gaps] -------|
```

**Self-RAG**: The LLM reflects on whether retrieved chunks are relevant before generating.

### Context Window Optimization

| Strategy | Description |
|---|---|
| **Sliding window** | Keep recent k chunks, discard older |
| **Relevance filtering** | Score each chunk, discard below threshold |
| **Compression** | Ask LLM to summarize retrieved chunks |
| **Structured extraction** | Extract only relevant facts, omit filler |
| **Lost-in-the-middle** | Place most relevant chunks at the **beginning** of the context |

The **Lost-in-the-Middle** paper (Liu et al., 2023) shows LLMs strongly prefer information at the start and end of the context window. **Always** order retrieved chunks by relevance descending.

```typescript
// Bad: chronological order
context = chunks.map(c => c.text).join("\n");

// Good: relevance-first order
context = chunks.sort((a, b) => b.relevance - a.relevance)
  .map(c => c.text).join("\n");
```

### Citation Strategies

For answer traceability — essential in production RAG:

```typescript
// Assign IDs to each chunk and ask LLM to cite them
const context = chunks.map((c, i) => `[${i}] ${c.text}`).join("\n\n");

const prompt = `
Answer the question using the context below.
Cite sources using [0], [1], etc. at the end of each sentence.

Context:
${context}

Question: ${query}
Answer:`;

// Expected output:
// "RAG stands for Retrieval-Augmented Generation [1]. It was introduced in 2020 [0]."
```

**Approaches**:
| Method | How | Tradeoff |
|---|---|---|
| Source tags | `[Doc:3]` in generated text | Simple but LLM may hallucinate numbers |
| Contains check | LLM answer token -> check which chunk contains it | Post-hoc, no hallucination prevention |
| Sentence-level | LLM answers sentence by sentence with citations | Slow but precise |

## 5. RAG Experiment: Key Takeaways

See `rag-experiment.ts` for a runnable end-to-end illustration.

The critical insight: **RAG is only as good as its weakest link**. A bad chunker destroys retrieval, a bad retriever gives the LLM bad context, and a bad prompt wastes good context. Tune all three in concert, starting with manual inspection of retrieved chunks before adding the LLM step.
